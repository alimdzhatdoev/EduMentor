# Дипломная работа

## Тема: «Разработка веб-приложения для автоматизации работы частного репетитора»

---

## Аннотация

Дипломная работа посвящена проектированию и разработке полнофункционального веб-приложения **EduMentor**, предназначенного для автоматизации деятельности частного репетитора. Система решает задачи управления учеником от момента первичной заявки до учёта оплат: ведение базы учеников, планирование занятий, назначение домашних заданий, хранение учебных материалов и контроль финансов. Дополнительно реализован публичный лендинг с формой онлайн-заявки для привлечения новых клиентов.

Приложение построено на современном технологическом стеке: React + TypeScript (фронтенд) и Node.js + Express + PostgreSQL (бэкенд). Система поддерживает два типа пользователей — репетитор и ученик — с разграниченными интерфейсами и уровнями доступа.

---

## 1. Введение

### 1.1 Актуальность темы

Рынок частного репетиторства продолжает активно расти. Репетиторы вынуждены вести административную работу вручную: записывать расписание в бумажные тетради или таблицы Excel, отслеживать оплаты по записям в телефоне, пересылать домашние задания через мессенджеры. Такой подход неэффективен, приводит к ошибкам и занимает значительное время, которое могло бы быть потрачено на непосредственную педагогическую деятельность.

Существующие системы управления учебным процессом (LMS) ориентированы на образовательные учреждения и избыточно сложны для индивидуального репетитора. Специализированных лёгких решений для частного специалиста практически нет.

Таким образом, разработка компактного и удобного веб-приложения, автоматизирующего рутинные задачи репетитора, является актуальной практической задачей.

### 1.2 Цель и задачи работы

**Цель:** разработать веб-приложение, автоматизирующее административную и организационную деятельность частного репетитора.

**Задачи:**
1. Проанализировать предметную область и требования к системе.
2. Спроектировать архитектуру клиент-серверного веб-приложения.
3. Спроектировать схему реляционной базы данных.
4. Разработать REST API на Node.js + Express с авторизацией на основе JWT.
5. Разработать пользовательский интерфейс на React с TypeScript.
6. Реализовать ролевую модель доступа (репетитор / ученик).
7. Разработать публичный лендинг с формой заявки.
8. Протестировать систему и подготовить документацию.

### 1.3 Объект и предмет исследования

**Объект:** процесс организации и ведения учёта в деятельности частного репетитора.

**Предмет:** веб-технологии и методы разработки информационных систем для автоматизации административных процессов.

## 2. Анализ предметной области

### 2.1 Бизнес-процессы репетитора

Деятельность частного репетитора включает следующие регулярные процессы:

1. **Привлечение учеников** — реклама услуг, получение заявок, коммуникация с потенциальными клиентами.
2. **Ведение базы учеников** — хранение контактных данных, уровня подготовки, предмета, индивидуальных заметок.
3. **Планирование расписания** — согласование и учёт времени занятий.
4. **Проведение занятий** — фиксация темы, формата (онлайн/офлайн), статуса занятия.
5. **Назначение домашних заданий** — выдача и контроль выполнения.
6. **Предоставление материалов** — отправка учебных файлов, ссылок и видео.
7. **Учёт оплат** — выставление счетов, отслеживание задолженностей.

### 2.2 Проблемы без автоматизации

| Процесс | Проблема |
|---|---|
| Расписание | Риск накладок, необходимость ручного отслеживания |
| Учёт оплат | Забытые должники, ошибки при ручном подсчёте |
| Домашние задания | Неудобная пересылка через мессенджеры, потеря истории |
| Материалы | Рассредоточены по разным папкам и чатам |
| Заявки | Теряются в переписке, нет единой базы |

### 2.3 Обзор аналогов

| Система | Преимущества | Недостатки для репетитора |
|---|---|---|
| Google Таблицы | Бесплатно, знакомо | Нет специализации, нет кабинета ученика |
| Moodle | Полнофункциональная LMS | Избыточна, сложна в настройке |
| Teachable / Thinkific | Продажа курсов | Не подходит для индивидуальных занятий |
| EduMentor (данная работа) | Специализация, личный кабинет ученика, простота | — |

## 3. Требования к системе

### 3.1 Функциональные требования

**Для репетитора (TUTOR):**
- Авторизация в системе
- Просмотр сводной статистики на дашборде
- Просмотр и обработка входящих заявок
- Управление базой учеников (добавление, просмотр, удаление)
- Планирование и учёт занятий (создание, редактирование, изменение статуса)
- Просмотр расписания в виде календаря по дням
- Назначение домашних заданий ученикам
- Загрузка и организация учебных материалов
- Учёт оплат и просмотр задолженностей

**Для ученика (STUDENT):**
- Авторизация в системе
- Просмотр личного дашборда с актуальной информацией
- Просмотр расписания занятий
- Просмотр истории занятий
- Просмотр и обновление статуса домашних заданий
- Доступ к учебным материалам
- Просмотр истории оплат

**Публичная часть (без авторизации):**
- Главная страница с описанием услуг
- Страница о репетиторе
- Страница услуг и предметов
- Страница цен
- Страница отзывов
- Контактная информация
- Форма онлайн-заявки

### 3.2 Нефункциональные требования

- **Безопасность:** хэширование паролей (bcrypt), JWT-авторизация, HTTP-заголовки безопасности (Helmet), CORS-политика.
- **Производительность:** SPA с клиентской маршрутизацией, минимизация запросов к API.
- **Удобство:** адаптивный интерфейс, современный дизайн (Tailwind CSS + shadcn/ui).
- **Масштабируемость:** модульная архитектура, разделение фронтенда и бэкенда.
- **Типобезопасность:** TypeScript на обоих уровнях приложения.

## 4. Проектирование системы

### 4.1 Архитектура

Приложение построено по архитектурному паттерну **клиент-сервер** с разделением на три уровня:

```
Уровень представления  →  React SPA (TypeScript + Vite)
Уровень логики         →  Node.js + Express (REST API)
Уровень данных         →  PostgreSQL + Prisma ORM
```

Такой подход обеспечивает:
- Независимость фронтенда и бэкенда (можно развернуть раздельно)
- Возможность подключения мобильного приложения к тому же API
- Чёткое разграничение зон ответственности

### 4.2 Ролевая модель

В системе определены две роли:

| Роль | Описание | Доступ |
|---|---|---|
| `TUTOR` | Репетитор | Полный доступ к управлению |
| `STUDENT` | Ученик | Просмотр своих данных, обновление статуса ДЗ |

Авторизация реализована через **JSON Web Token (JWT)** с временем жизни 7 дней. Токен хранится в `localStorage` браузера и передаётся в заголовке каждого запроса к API.

### 4.3 Схема базы данных

База данных содержит 6 основных сущностей:

```
User           — пользователи системы (репетитор и ученики)
Application    — заявки от потенциальных учеников
Student        — расширенный профиль ученика (1:1 с User)
Lesson         — занятия (N:1 к Student)
Homework       — домашние задания (N:1 к Student, N:1 к Lesson)
Material       — учебные материалы (N:1 к Student или общие)
Payment        — платёжные записи (N:1 к Student)
```

Все первичные ключи — CUID (криптографически случайные строковые идентификаторы), что обеспечивает уникальность и безопасность.

### 4.4 API Design

API следует принципам REST:
- Ресурсы именуются существительными во множественном числе (`/students`, `/lessons`)
- HTTP-методы отражают действия: `GET` (чтение), `POST` (создание), `PATCH` (обновление), `DELETE` (удаление)
- Ответы в формате JSON
- Статусные коды: `200 OK`, `201 Created`, `400 Bad Request`, `401 Unauthorized`, `403 Forbidden`, `404 Not Found`, `500 Internal Server Error`

## 5. Реализация

### 5.1 Серверная часть (Backend)

**Технологический стек:** Node.js 18+, TypeScript, Express 4, Prisma ORM 5, PostgreSQL, JWT, bcrypt.

Сервер организован по принципу маршрутизатора (`Router`). Каждый модуль данных имеет собственный файл маршрутов:

```
src/
├── index.ts           # Инициализация Express, подключение роутеров
├── middleware/
│   └── auth.ts        # JWT-аутентификация, requireTutor
└── routes/
    ├── auth.ts        # POST /login, GET /me
    ├── applications.ts
    ├── students.ts
    ├── lessons.ts
    ├── homework.ts
    ├── materials.ts
    └── payments.ts
```

Ключевые технические решения:
- **Prisma ORM** позволяет работать с базой данных через типобезопасный TypeScript API, автоматически генерирует клиент по схеме.
- **bcryptjs** используется для необратимого хэширования паролей (salt rounds = 10).
- **Helmet** устанавливает защитные HTTP-заголовки (`Content-Security-Policy`, `X-Frame-Options` и др.).
- **CORS** настроен только для домена фронтенда (`http://localhost:5173`).
- Разграничение доступа для занятий, заданий и платежей: репетитор видит все записи, ученик — только свои (фильтрация по `userId` через связь с `Student`).

### 5.2 Клиентская часть (Frontend)

**Технологический стек:** React 18, TypeScript, Vite, React Router 6, Tailwind CSS, shadcn/ui, Zustand, Axios, React Hook Form, Zod.

Приложение организовано по функциональному принципу:

```
src/
├── App.tsx             # Дерево маршрутов
├── layouts/            # Общие оболочки страниц
├── pages/
│   ├── public/         # 7 публичных страниц лендинга
│   ├── dashboard/      # 9 страниц кабинета репетитора
│   └── student/        # 6 страниц кабинета ученика
├── components/         # Переиспользуемые UI-компоненты
├── store/authStore.ts  # Глобальное состояние авторизации
├── lib/api.ts          # Настроенный axios с интерсепторами
└── types/index.ts      # TypeScript-типы всех сущностей
```

**Управление состоянием:** Zustand используется исключительно для хранения данных авторизации (`token`, `user`). Данные каждой страницы загружаются локально в хуках `useEffect`, без избыточного глобального состояния.

**Формы:** React Hook Form + Zod обеспечивают декларативную валидацию форм с типобезопасными схемами.

**Маршрутизация:** React Router 6 с вложенными маршрутами. `ProtectedRoute` защищает приватные разделы: проверяет наличие токена и роли, иначе перенаправляет на `/login`.

**Два макета:**
- `PublicLayout` — шапка с навигацией и подвал для лендинга
- `DashboardLayout` — боковое меню + основная зона контента для личных кабинетов

### 5.3 Интерфейс пользователя

Дизайн построен на компонентной библиотеке **shadcn/ui** (основана на Radix UI) и утилитарном CSS-фреймворке **Tailwind CSS**. Это обеспечивает:
- Доступность (accessibility) из коробки через Radix UI
- Консистентный внешний вид
- Быструю разработку без написания CSS вручную

Ключевые UI-паттерны:
- `StatusBadge` — цветные бейджи для отображения статусов (занятий, заданий, оплат)
- `EmptyState` — заглушка при пустых списках
- `PageHeader` — единый заголовок страниц с кнопкой действия
- Модальные диалоги (Radix Dialog) для форм создания/редактирования

Для проверки функциональности разработан seed-скрипт (`prisma/seed.ts`), который автоматически создаёт:

- Домашние задания с разными статусами
- Учебные материалы разных типов
- Платёжные записи (оплаченные и задолженности)
- Тестовые заявки с разными статусами

### 6.2 Тестовые сценарии

| Сценарий | Ожидаемый результат |
|---|---|
| Вход с верными данными репетитора | Перенаправление на `/dashboard` |
| Вход с верными данными ученика | Перенаправление на `/student` |
| Вход с неверным паролем | Сообщение об ошибке, остаёмся на `/login` |
| Ученик пытается открыть `/dashboard` | Перенаправление на `/login` |
| Создание ученика репетитором | Ученик появляется в списке, создаётся аккаунт |
| Ученик меняет статус ДЗ | Статус обновляется, виден репетитору |
| Подача заявки без авторизации | Заявка сохраняется, видна в кабинете репетитора |
| Истечение JWT-токена | Автоматический редирект на `/login` |
